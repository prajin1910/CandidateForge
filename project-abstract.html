Abstract

CandidateForge is a browser-native ETL (Extract, Transform, Load) application built with React.js that consolidates fragmented candidate data from multiple sources into a single, mathematically verified Canonical Profile. In modern HR workflows, candidate information is spread across structured ATS databases (JSON/CSV), uploaded résumé documents (PDF), and live developer profiles (GitHub). CandidateForge extracts data from all three sources in parallel, normalises every field to strict international standards, resolves conflicts using a user-defined priority hierarchy, and attaches confidence scores and full provenance metadata to each field. The entire pipeline runs client-side with zero backend dependency, powered by the GitHub REST API and a pdf.js WASM worker for document parsing.

System Architecture & Data Flow

Problem Statement

Organisations manage candidate profiles across disconnected systems — ATS databases (JSON/CSV), uploaded PDF résumés, and GitHub developer profiles. Merging these manually is error-prone, leading to duplicated and inconsistent records. CandidateForge automates this process deterministically through a structured pipeline.

10-Stage Transformation Pipeline

  1. Config Loader — reads user-defined pipeline settings
  2. Source Detection — identifies available input types
  3. Parallel Extraction — JSON parser, GitHub API, pdf.js run concurrently
  4. Field Normalisation — dates (ISO-8601), phones (E.164), skills (canonical)
  5. Source Aggregation — merges all extracted records into one object
  6. Conflict Resolution — user priority order; highest non-null value wins
  7. Confidence Scoring — cross-source agreement boosts score from 0.7 to 1.0
  8. Provenance Builder — attaches source, score, and timestamp to each field
  9. Canonical Builder — assembles the final unified profile object
  10. Schema Validation — validates output against the canonical JSON schema

Key Features

  • Parallel Extraction — GitHub REST API, pdf.js WASM worker, and JSON parser execute concurrently in the browser event loop.
  • Conflict Resolution Engine — drag-and-drop source priority hierarchy; field-level winner selection is deterministic.
  • Confidence Scoring — trust score per field: 0.70 (single source) up to 1.0 (all sources agree).
  • Full Provenance Chain — every output field carries { value, source, confidence, timestamp }.
  • Live Telemetry — real-time pipeline log stream in the UI as each stage executes.
  • Automated Validation Suite — test runner at /validation for duplicates, null fallbacks, and schema failures.
  • Zero Backend — fully client-side; deployed as a static site on Vercel.

Application Flow

The user navigates a 5-step wizard: (1) upload structured JSON/CSV, (2) select additional sources, (3) provide the GitHub URL or PDF file, (4) configure pipeline rules and source priority order, (5) trigger the transform and view real-time logs. The output screen displays the Canonical Profile with confidence badges, a conflict log, and a provenance map for every field.

Testing & Validation

The /validation route runs an automated test suite verifying: Gold Profile Comparison (accuracy against expected outputs), Duplicate Skill Detection (canonicalisation correctness), and Missing Value Fallbacks (graceful handling of failed API calls or blank PDFs).
